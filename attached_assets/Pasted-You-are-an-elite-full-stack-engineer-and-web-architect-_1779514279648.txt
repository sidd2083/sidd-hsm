### 🧮 4. THE MATHEMTICAL ENGINE & PARIMUTUEL POOL LOGIC
Do NOT use a lazy fixed-odds system or a simple double-your-money mechanic. You must implement a proper Parimutuel Betting Pool Engine where payout multipliers swing dynamically based on public market volume.

1. Live Probability & Multiplier Formulas:
   - Total Market Pool = yesPool + noPool
   - YES Multiplier = Total Market Pool / yesPool (Expressed live on-screen as an "X" value, e.g., "1.34x")
   - NO Multiplier = Total Market Pool / noPool (Expressed live on-screen as an "X" value, e.g., "4.15x")
   - The UI must calculate and render these numbers in real time as money pools shift. If a pool is brand new and empty, default the calculations to an even "2.00x" split.

2. Fixed Stake Proportional Payout Logic:
   The system must calculate historical returns based on the user's specific pool stake percentage *at the time they placed their bet*.
   - When a user bets, they are buying a portion of that specific side's pool.
   - Payout Algorithm: User Payout = (User's Amount Bet / Total Cash in that specific pool at the time of purchase) * Total Combined Final Pool Cash.
   - Example Impact: If 90% of the school bets on Spain (YES) and only 10% bets on Portugal (NO), the YES multiplier drops near 1.1x while the NO multiplier climbs to 10.0x. If Portugal pulls off an upset and the admin resolves the market as NO, the platform seizes the entire Rs. 90,000 Spain pool and distributes it proportionally among the Portugal backers. Someone who bet Rs. 1,000 on Portugal early walks away with a massive Rs. 10,000 cash out.

3. Automated Server-Side Time-Lock Protocol:
   Every single market document contains a `lockTimestamp` (Unix epoch milliseconds). The millisecond a user clicks a button to execute a bet transaction, the script must perform an instant evaluation check:
   - If (Date.now() >= market.lockTimestamp), instantly block the transaction, reject the trade, disable the interactive UI buttons, and flag the card with a clear, solid, visual `🔒 Locked` status badge.

4. The 24-Hour Engagement Stimulus:
   Implement an automated daily allowance check. The first time a logged-in user hits the platform past 06:00 AM each day, verify if `lastDailyClaimDate != currentLocalDate`. If true, execute an atomic Firestore transaction to update the date field and instantly credit `+Rs. 100` straight into their profile balance.
