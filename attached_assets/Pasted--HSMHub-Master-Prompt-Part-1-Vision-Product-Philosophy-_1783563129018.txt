# HSMHub Master Prompt – Part 3 (Condensed)

## Architecture, Performance, Firestore, Security, Admin Dashboard & Production Standards

Continue from Part 1 and Part 2. Do not redesign previously approved features. Preserve the existing Prediction Market logic while implementing the following engineering and architecture requirements.

---

# ARCHITECTURE

HSMHub must be built using a modular architecture.

Each feature should be independent:

* Prediction Market
* HSM Newz
* Confessions
* Replies
* Wallet
* Notifications
* Profile
* Admin Dashboard

Each module should have its own services, UI components, validation, and business logic. Avoid tightly coupling unrelated features.

---

# DATABASE DESIGN

Organize data into clear collections (or equivalent structures):

* Users
* Wallet Transactions
* Prediction Markets
* Bets
* News
* Confessions
* Replies
* Notifications
* Reports
* Settings

Store images outside the database (for example, in cloud storage) and keep only image URLs in the database.

Every document should include:

* Created At
* Updated At
* Created By (internal reference)
* Status
* Unique ID

---

# PERFORMANCE

The application should feel instant.

Requirements:

* Lazy load images.
* Paginate all feeds.
* Load 20 items initially.
* Load more only when needed.
* Avoid loading the entire collection.
* Minimize unnecessary re-renders.
* Optimize bundle size.
* Compress uploaded images.
* Use skeleton loaders while content loads.

---

# FIRESTORE OPTIMIZATION

Design to reduce cost and improve scalability.

* Never fetch an entire collection.
* Fetch only required fields where possible.
* Use pagination for feeds.
* Avoid unnecessary real-time listeners.
* Use indexes for common queries.
* Cache data when appropriate.
* Store only URLs for uploaded images.
* Archive or delete old rejected content after a retention period if it is no longer needed.

Important: Voting cannot be implemented with **zero reads and zero writes**. Every persisted vote must update some state. The goal is to minimize operations, not eliminate them entirely.

---

# VOTING SYSTEM

Each authenticated user can vote only once per post.

Allow:

* Upvote
* Downvote

Users can change or remove their vote.

Prevent duplicate voting.

Do not trust client-side checks alone.

Use efficient aggregation strategies so the feed does not need to recalculate every vote on each page load.

---

# ANTI-SPAM

Prevent abuse by:

* Rate limiting submissions.
* Preventing duplicate content.
* Limiting repeated actions within a short period.
* Rejecting oversized uploads.
* Logging suspicious activity.
* Allowing administrators to suspend abusive accounts.

---

# WALLET SECURITY

All wallet changes must be performed by trusted backend logic.

Clients must never be able to:

* Increase balance.
* Edit transaction history.
* Reward themselves.
* Refund themselves.

Every balance change must create a transaction log with:

* Amount
* Reason
* Timestamp
* Related entity (market, news, confession, reply)
